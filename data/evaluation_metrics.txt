Evaluation Metrics for RAG Pipelines

Evaluating a RAG pipeline requires measuring two separate subsystems: the retriever (did it find the right chunks?) and the generator (did it produce a good answer?). Each subsystem has its own set of metrics.

Retrieval Evaluation Metrics

Precision@K measures the fraction of the K retrieved chunks that are actually relevant to the query.

    Precision@K = |relevant ∩ retrieved| / K

A Precision@3 of 1.0 means all three retrieved chunks are relevant. A value of 0.33 means only one of three was relevant. High precision indicates the retriever is not cluttering the LLM context with irrelevant passages.

Recall@K measures the fraction of all relevant chunks that appear in the top-K results.

    Recall@K = |relevant ∩ retrieved| / |relevant|

Recall matters when a query requires synthesizing information from multiple relevant chunks. Low recall means some necessary context was missed.

MRR (Mean Reciprocal Rank) is the average of the reciprocal rank of the first relevant result across all queries.

    MRR = mean(1 / rank_of_first_relevant)

If the first relevant chunk appears at rank 1, MRR contribution is 1.0. At rank 2, it is 0.5. At rank 3, it is 0.33. MRR rewards systems that place a relevant result near the top of the ranking.

Hit@K (also called Hit Rate) is a binary metric: 1 if at least one relevant chunk appears in the top-K results, 0 otherwise. Hit Rate is the mean of Hit@K across all queries. It answers the question "for what fraction of queries did the retriever find at least one useful chunk?"

Generation Evaluation Metrics

Faithfulness measures whether every claim in the generated answer is supported by the retrieved context. An answer that introduces facts not present in the context is hallucinating. Faithfulness is scored 0.0 to 1.0, where 1.0 means fully grounded.

Answer Relevance measures whether the generated answer actually addresses the question that was asked. A highly faithful but off-topic answer scores low on relevance. Scored 0.0 to 1.0.

Answer Correctness measures how well the generated answer matches a reference (ground-truth) answer. It combines factual overlap and semantic similarity. Requires a golden dataset with reference answers. Scored 0.0 to 1.0.

LLM-as-Judge Evaluation

Because generation quality is difficult to measure with rule-based metrics, the standard approach is LLM-as-judge: a separate LLM (the judge) reads the query, context, and generated answer and produces numerical scores for each metric along with a reasoning explanation.

The judge prompt must be carefully designed to elicit consistent, calibrated scores. Using a strong judge model (such as claude-opus-4-8 with adaptive thinking) improves scoring reliability. Scores should be clamped to [0.0, 1.0] and validated before aggregation.

Aggregation

Per-query metrics are aggregated to dataset-level metrics using the macro average: sum all per-query scores and divide by the number of queries. Macro averaging treats every query equally regardless of the number of relevant chunks it has.

RAGAS Framework

RAGAS (Retrieval Augmented Generation Assessment) is an open-source evaluation framework that implements faithfulness, answer relevance, context precision, and context recall as automated metrics using LLM judges. It provides a standard benchmark suite and supports integration with popular RAG frameworks.

Evaluation-Driven Development

In evaluation-driven development, a golden dataset of query-answer pairs is created before the RAG pipeline is built. The evaluation suite is run after every configuration change — chunk size, top_k, embedding model, prompt template — so that regressions are caught immediately and improvements can be measured objectively.
